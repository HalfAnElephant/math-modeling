# C题三阶段研究路线图：多无人机-物业人员联合巡检优化

## 0. 问题总体分析

C题研究对象是智慧社区中的“无人机空中巡检—物业人员现场复核”闭环巡检机制。社区内共有 16 个巡检目标点，所有无人机从节点 0 出发并返回节点 0；每架无人机可以在运营时段内执行多趟任务，每趟任务受到单次有效电量上限约束，返航后需要更换电池。无人机到达目标点后需要悬停观测，目标点的累计悬停时间达到基础要求时，视为完成基础空中巡检；累计悬停时间达到更高的直接确认阈值时，视为可由无人机直接确认，否则需要物业人员在无人机阶段结束后统一出发进行现场复核。

本题不是单纯的最短路或旅行商问题，而是一个由多类决策耦合形成的组合优化问题：

1. 多无人机任务分配：决定目标点由哪些无人机、哪些趟次巡检。
2. 多趟路径规划：决定每趟无人机从节点 0 出发后访问哪些目标点、按什么顺序访问、再返回节点 0。
3. 悬停时间分配：决定每个目标点累计悬停时间是否只满足基础巡检，还是进一步满足直接确认。
4. 能耗与时间约束：每趟任务不能超过有效电量上限，所有无人机阶段任务应在运营时段内完成。
5. 空地协同闭环：若无人机不直接确认，物业人员需要在地面路径上复核这些点。

因此，整体研究思路应采用“数学规划建模为主，图论路径优化和启发式算法为辅”的路线。主模型必须保证约束清晰、解释性强、结果可复现；算法用于在有限时间内搜索高质量可行解。前面已经判断过，本题暂不引入 ML 或 DL，因为数据只有一个确定性场景实例，缺少历史样本和标注结果。若硬引入机器学习，容易造成数据依据不足、黑箱解释困难、可行性约束难保证等副作用。

整体推荐路线分为三个阶段：

- 第一阶段：数据理解与基础建模准备。
- 第二阶段：问题1基础空中巡检优化。
- 第三阶段：问题2空地联合闭环优化与结果分析。

三个阶段之间是递进关系：第一阶段建立数据、符号、评价指标和可行性边界；第二阶段求解无人机完成基础巡检的最优或近优方案；第三阶段在第二阶段基础上加入直接确认和物业复核，形成最终闭环方案和配置建议。

### 1.4 符号体系

建议在论文中统一使用以下符号：

- `S={1,2,...,n}`：巡检目标点集合，`n=16`。
- `0`：无人机起降服务中心。
- `P0`：物业人员出发点，与节点 0 同址。
- `K`：无人机数量。
- `r`：无人机单次出航任务编号。
- `i,j`：无人机路径节点。
- `s`：巡检目标点。
- `t^f_{ij}`：无人机从节点 `i` 到节点 `j` 的飞行时间。
- `e^f_{ij}`：无人机从节点 `i` 到节点 `j` 的飞行能耗。
- `g_{ij}`：物业人员在复核点 `i,j` 之间的地面通行时间。
- `b_s`：目标点 `s` 的基础悬停时间要求，即 `base_hover_time_s`。
- `h_s`：目标点 `s` 的无人机直接确认阈值，即 `direct_confirm_time_s`。
- `w_s`：目标点 `s` 的优先级权重。
- `T_s`：目标点 `s` 的累计悬停时间。
- `C_s`：目标点 `s` 的空中巡检完成时刻。
- `T_u`：无人机阶段完成时间。
- `T_g`：物业复核阶段耗时。
- `T=T_u+T_g`：闭环总完成时间。
- `E_max`：单趟有效能耗上限。
- `P_h`：无人机悬停功率。
- `tau_b`：换电时间。

### 1.5 第一阶段解决方案

第一步，建立数据字典。逐项列出每个工作表字段含义，尤其要明确 `base_hover_time_s` 与 `direct_confirm_time_s` 的区别。前者是所有点必须满足的基础空中巡检要求，后者是无人机直接确认所需的更高累计悬停阈值。

第二步，建立时间与能耗计算公式。对任意一趟无人机任务，若路径为：

```text
0 -> i1 -> i2 -> ... -> im -> 0
```

则任务总时间为：

```text
route_time = sum(t^f_{ij}) + sum(hover_s)
```

任务总能耗为：

```text
route_energy = sum(e^f_{ij}) + P_h * sum(hover_s)
```

必须满足：

```text
route_energy <= E_max
```

第三步，建立无人机总作业时间。若某无人机执行 `R_k` 趟任务，则其总时间为：

```text
work_time_k = sum(route_time_{k,r}) + (R_k - 1) * tau_b
```

无人机阶段完成时间为：

```text
T_u = max_k work_time_k
```

第四步，建立物业复核时间。若待复核点集合为 `M`，物业人员路径为：

```text
P0 -> m1 -> m2 -> ... -> mq -> P0
```

则物业复核阶段耗时为：

```text
T_g = sum(ground_time_edges) + sum(manual_service_time_s)
```

第五步，形成阶段性结论。通过基础统计和简单下界说明：问题1重点是多无人机协同和多趟能耗切分；问题2重点是直接确认阈值较高造成的空地替代决策。

## 第二阶段：问题1多无人机基础空中巡检优化

### 2.1 阶段目标

问题1要求在所有目标点都完成基础空中巡检的前提下，确定多架无人机的任务分配、出航次数、访问顺序和各点悬停时间，使系统总巡检完成时间尽可能短，同时兼顾高优先级点提前完成、总能耗和负载均衡。

本阶段完成后，应能够给出：

- 对每个 `K=1,2,3,4` 的基础巡检方案。
- 每架无人机执行几趟任务。
- 每趟任务访问哪些目标点及顺序。
- 每个目标点的累计悬停时间。
- 每个方案的完成时间、总能耗和负载均衡指标。
- 换电时间 `tau_b` 对巡检效率的影响。
- 推荐无人机配置规模及边际收益解释。

### 2.2 问题1建模思路

问题1可以抽象为“多无人机、多趟次、带单趟能耗约束的车辆路径问题”。与标准 VRP 不同的是，每个点的服务时间不是固定由一次访问完成，而是可以由多架无人机、多趟任务累计完成。由于问题1只要求基础巡检，最简洁且稳健的设定是让每个点累计悬停时间刚好等于 `b_s`，除非后续局部优化中拆分访问会带来更优路径。

决策变量建议包括：

- `x_{krij}`：无人机 `k` 第 `r` 趟任务是否从节点 `i` 飞到节点 `j`。
- `y_{krs}`：无人机 `k` 第 `r` 趟是否访问目标点 `s`。
- `u_{krs}`：无人机 `k` 第 `r` 趟在目标点 `s` 的悬停时间。
- `C_s`：目标点 `s` 的基础巡检完成时刻。
- `L_k`：无人机 `k` 的总作业时长。
- `T_u`：无人机阶段完成时间。

核心约束：

```text
sum_{k,r} u_{krs} >= b_s                       对每个目标点 s
sum(flight_energy) + P_h * sum(hover_time) <= E_max   对每一趟任务
每趟任务从 0 出发，并最终回到 0
路径满足流守恒和子回路消除
L_k = sum(route_time_{k,r}) + (R_k - 1) * tau_b
T_u >= L_k                                      对每架无人机 k
T_u <= operating_horizon_s
```

目标函数采用多目标加权：

```text
min F1 = alpha*T_u + beta*sum(w_s*C_s) + gamma*E_total + delta*LoadImbalance
```

推荐权重：

```text
alpha = 0.55
beta  = 0.20
gamma = 0.15
delta = 0.10
```

解释：完成时间是主目标，高优先级提前完成和能耗是次目标，负载均衡用于避免某一架无人机承担过多任务。

### 2.3 问题1求解方案

由于目标点只有 16 个，规模不大，但多无人机、多趟、能耗、换电和服务时间耦合后，直接写完整 MILP 会比较复杂。建议采用“任务列生成思想 + 启发式优化”的求解方式。

第一步，构造初始访问顺序。可以从服务中心出发，使用最近邻法或按空间聚类生成初始顺序。目标是让相邻目标点空间距离较近，降低飞行时间和飞行能耗。

第二步，按能耗上限切分任务。沿初始顺序逐个加入目标点，计算当前单趟任务的飞行能耗和基础悬停能耗。如果加入下一个点会超过 `E_max`，则结束当前任务，开启新一趟任务。

第三步，对每趟任务进行 2-opt 局部优化。固定每趟任务包含的目标点集合，对访问顺序做反转交换，只接受能降低任务时间或能耗且仍满足能耗约束的路径。

第四步，将任务分配给 `K` 架无人机。按任务耗时从大到小排序，每次把当前任务分配给累计工作时间最短的无人机，分配时计入换电时间。

第五步，进行负载均衡调整。尝试把某一趟任务从负载最重的无人机移动到负载较轻的无人机，若 `T_u` 降低则接受调整。

第六步，计算全部指标。包括 `T_u`、`E_total`、负载标准差、高优先级完成时刻加权和、每架无人机任务数、每趟任务能耗。

### 2.4 问题1实验设计

问题1至少应完成三组实验。

第一组：无人机数量对比。

```text
K = 1, 2, 3, 4
```

对比指标：

- 无人机阶段完成时间 `T_u`。
- 总飞行与悬停能耗 `E_total`。
- 每架无人机工作时长。
- 负载均衡指标。
- 任务趟次数。

第二组：换电时间敏感性。

```text
tau_b = 0, 150, 300, 450, 600
```

固定推荐或基准无人机数量，例如 `K=3`，观察换电时间增加时 `T_u` 的变化。若任务趟数较多，换电时间影响会更明显。

第三组：边际收益递减分析。

计算：

```text
Delta(K) = T_u(K-1) - T_u(K)
```

当从 `K=3` 增加到 `K=4` 的时间改善显著小于从 `K=1` 到 `K=2` 或 `K=2` 到 `K=3` 的改善时，可以说明无人机配置存在边际收益递减，从而给出推荐规模。

## 第三阶段：问题2空地联合闭环优化与最终建议

### 3.1 阶段目标

问题2在问题1基础上进一步考虑无人机直接确认和物业人工复核。核心是判断：对于每个目标点，是让无人机增加悬停时间达到直接确认阈值，还是只完成基础巡检后交给物业人员复核。

本阶段完成后，应能够给出：

- 哪些点由无人机直接确认。
- 哪些点需要物业人员复核。
- 多无人机联合巡检路线和物业复核路线。
- 闭环总完成时间 `T=T_u+T_g`。
- 不同无人机数量下的闭环效率对比。
- 不同直接确认阈值下的人工复核工作量变化。
- 推荐无人机数量和阈值设定建议。

### 3.2 问题2建模思路

在问题1的变量基础上，引入二元变量：

```text
z_s = 1  表示目标点 s 由无人机直接确认
z_s = 0  表示目标点 s 需要物业复核
```

目标点的累计悬停时间仍为：

```text
T_s = sum_{k,r} u_{krs}
```

直接确认约束为：

```text
T_s >= h_s * z_s
```

基础巡检约束仍然保留：

```text
T_s >= b_s
```

若 `z_s=0`，则目标点 `s` 进入物业复核集合 `M`。物业人员在无人机阶段完成后，从 `P0` 出发访问所有待复核点并返回 `P0`，形成地面 TSP：

```text
P0 -> m1 -> m2 -> ... -> mq -> P0
```

物业复核耗时为：

```text
T_g = ground_travel_time + manual_service_time
```

闭环总时间为：

```text
T = T_u + T_g
```

目标函数建议设置为：

```text
min F2 = lambda1*(T_u + T_g)
       + lambda2*sum(w_s*(1-z_s))
       + lambda3*sum(1-z_s)
       + lambda4*E_total
       + lambda5*LoadImbalance
```

推荐权重：

```text
lambda1 = 0.50
lambda2 = 0.20
lambda3 = 0.10
lambda4 = 0.10
lambda5 = 0.10
```

解释：闭环总时间是主目标；第二项强调高优先级点尽量不要转人工；第三项控制人工复核点数量；第四、五项分别控制能耗和无人机负载均衡。

### 3.3 空地替代收益思想

问题2最重要的创新点是提出“空地替代收益”。对每个目标点，直接确认需要付出的额外无人机代价为：

```text
extra_hover_s = h_s - b_s
extra_hover_energy = P_h * extra_hover_s
```

如果该点不由物业复核，可以节省：

```text
manual_service_time_s + 该点对物业地面路径的边际通行时间
```

因此可构造一个解释性指标：

```text
B_s = w_s * manual_saving_s / (extra_hover_s + epsilon)
```

其中 `manual_saving_s` 可以用从当前物业路径中删除该点后减少的总复核时间估计，也可以用从 `P0` 往返该点的时间加现场服务时间作为初始近似。`B_s` 越大，说明该点越适合由无人机直接确认。

这个指标不替代优化模型，而是用于：

- 构造联合优化初始解。
- 解释哪些点优先由无人机直接确认。
- 支持论文中的管理建议。

### 3.4 问题2求解方案

第一步，继承问题1基础巡检方案。所有点先满足基础巡检要求，得到基础无人机路径和 `T_u`。

第二步，计算每个目标点的直接确认收益。对每个点计算 `extra_hover_s`、额外悬停能耗、物业复核节省时间和优先级权重，得到 `B_s`。

第三步，按 `B_s` 从高到低尝试升级目标点。所谓升级，就是在无人机方案中给该点增加悬停时间，使其累计悬停达到 `h_s`。可以采用两种方式：

- 如果原路线中已有该点，则在原访问处增加悬停时间。
- 如果增加后导致单趟能耗超限，则为该点新建一趟单点直接确认任务，或将其分配到其他有余量的任务中。

第四步，每次升级后重新计算无人机阶段完成时间、能耗和物业复核集合。若闭环总时间下降，或高优先级人工复核点数明显减少且时间代价可接受，则保留该升级。

第五步，对剩余物业复核点求解地面 TSP。由于最多 16 个点，可使用动态规划或枚举优化；若人工复核点较多，也可使用最近邻加 2-opt。

第六步，输出联合方案。包括无人机每趟任务、直接确认点集合、物业复核点集合、物业复核顺序和闭环总时间。

### 3.5 问题2实验设计

第一组：基础空中巡检与联合巡检对比。

对比方案：

- 方案 A：只完成基础空中巡检，所有未达到直接确认阈值的点交给物业。
- 方案 B：联合巡检，选择部分目标点由无人机直接确认。

对比指标：

- 无人机阶段时间 `T_u`。
- 物业复核时间 `T_g`。
- 闭环总时间 `T`。
- 人工复核点数量。
- 高优先级人工复核点数量。
- 无人机总能耗。

第二组：不同无人机数量下的闭环效率。

```text
K = 1, 2, 3, 4
```

观察无人机数量增加后：

- 闭环总时间是否下降。
- 直接确认比例是否上升。
- 人工复核点数是否下降。
- 是否存在边际收益递减。

第三组：直接确认阈值敏感性。

将所有 `direct_confirm_time_s` 统一乘以：

```text
0.70, 0.85, 1.00, 1.15, 1.30
```

观察阈值降低或升高对以下指标的影响：

- 无人机直接确认点数。
- 人工复核点数。
- 闭环总完成时间。
- 无人机能耗。

如果阈值过高，直接确认代价过大，更多点应交给物业；如果阈值适度降低，无人机直接确认比例提高，物业工作量减少，闭环效率可能改善。

### 3.6 第三阶段论文表达重点

问题2应突出“闭环效率”而非单纯无人机效率。只看无人机阶段时间，基础巡检方案可能最快；但加入物业复核后，总闭环时间可能并不最优。因此，必须比较 `T_u` 与 `T_u+T_g` 的差异。

论文中建议重点展示：

- 哪些目标点被无人机直接确认，为什么。
- 哪些目标点交给物业复核，为什么。
- 高优先级点是否优先被无人机确认。
- 直接确认阈值变化如何影响人工工作量。
- 推荐配置是否兼顾时间、能耗和人员负担。

### 3.7 第三阶段产出物

本阶段应产出：

- 问题2联合优化模型。
- 直接确认点与人工复核点列表。
- 物业复核 TSP 路径。
- 基础巡检方案与联合巡检方案对比表。
- 不同 `K` 下闭环效率对比表。
- 直接确认阈值敏感性表。
- 物业复核路径图。
- 闭环总时间、人工复核点数、直接确认比例变化图。
- 最终推荐无人机数量与阈值建议。

### 3.8 第三阶段检查点

完成本阶段后，应确认：

- 所有目标点仍满足基础空中巡检要求。
- 被标记为无人机直接确认的点确实满足 `T_s >= h_s`。
- 被标记为物业复核的点都出现在物业复核路径中。
- 物业路径从 `P0` 出发并回到 `P0`。
- 闭环总时间计算为 `T_u + T_g`。
- 推荐方案不是只追求无人机时间最短，而是综合闭环效率、人工负担、优先级和能耗。
